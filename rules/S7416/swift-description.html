Description

The release build of an application is generally meant for distribution to end users. By making this target debuggable, you expose your application to unnecessary risks. Debug instructions or error messages can leak detailed information about the system, like the applicationâ€™s path or file names.

Why is this an issue?

When you build a release version of your application, you’re creating a distributable package of your code. Making release builds debuggable will make it much easier for someone to reverse-engineer your application. This means they can potentially understand your code’s logic, extract sensitive information like API keys, or even modify your application for malicious purposes.

Release builds are meant for distribution to end users and is therefore under constant scrutiny. Making these builds debuggable creates a serious vulnerability. While debugging builds can be used for troubleshooting, failing to enable it for release builds exposes your application to unnecessary risks.

How to fix it in XCode

Code examples

Noncompliant code example

{
	/* ... */
	objects = {
		ABCDEF123456789ABCDFE123 /* Release */ = {
			isa = XCBuildConfiguration;
			buildSettings = {
				/* ... */
				SWIFT_ACTIVE_COMPILATION_CONDITIONS = "DEBUG"; /* Noncompliant */
				STRIP_INSTALLED_PRODUCT = NO; /* Noncompliant */
				ENABLE_TESTABILITY = YES; /* Noncompliant */
				/* ... */
			};
			name = Release;
		};
	};
}

This snippet of pbxproj has a misconfigured "Release" profile where:

  • Debug symbols will not be striped from the binary as

    • Enable Testability is set to "Yes"

    • Strip Linked Product is set to "No"

    • The DEBUG Active Compilation Condition is set

Compliant solution

{
	/* ... */
	objects = {
		ABCDEF123456789ABCDFE123 /* Release */ = {
			isa = XCBuildConfiguration;
			buildSettings = {
				/* ... */
				SWIFT_ACTIVE_COMPILATION_CONDITIONS = "";
				STRIP_INSTALLED_PRODUCT = YES;
				ENABLE_TESTABILITY = NO;
				/* ... */
			};
			name = Release;
		};
	};
}
